# Tab Harbor 要件定義

作成日: 2026-07-09

## 1. 目的

Arc から Edge への移行にあたり、「Pin 中心のサイドバー」「Space によるコンテキスト分離」
「Space⇄Profile 連動」の体験を Edge 上で可能な限り再現する。

## 2. 機能要件

### FR-1: サイドバー UI(`sidePanel`)

- サイドバー上部に Pin をアイコングリッド/リストで表示、下部に「現在開いているタブ」を
  表示する(Arc と同じ上下構造)
- タブの開閉・切替・タイトル変更をリアルタイム反映する
- ツールバーアイコンクリックまたはショートカット(`commands` API)で開閉できる

### FR-2: Pin(固定ブックマーク)

- URL を Pin として登録し、Space ごとに保持する。並べ替え・フォルダ分け可
- クリック時: 同じ URL のタブが開いていればそれにフォーカス、なければ新規タブで開く
  (Arc の「アプリ的」挙動の再現)
- データは `storage.sync` に保存し、同一プロファイルの Edge 間で同期する

### FR-3: ホバープレビュー

- Pin にホバーでポップアップを表示する。内容は:
  1. タイトル・URL・favicon(常に可能)
  2. 最後に訪問した際のスクリーンショットのサムネイル(キャッシュ方式)
- **制約:** Arc の Peek のようなライブ埋め込みプレビューは、多くのサイトが
  `X-Frame-Options` / CSP `frame-ancestors` で iframe 埋め込みを拒否するため保証できない。
  サムネイルキャッシュ方式を正とする

### FR-4: Space

- Space の作成・命名・アイコン/テーマ色設定。各 Space が「Pin セット+タブセット」を持つ
- Space 切替時に現在のタブ群を退避し、対象 Space のタブ群を復元する
  (実装方式は §4 の設計判断を参照)
- キーボードショートカットで Space を巡回切替できる

### FR-5: Profile 連動(条件付き)

- Space に「別プロファイルで開く」設定を持たせ、切替時に Native Messaging ホスト経由で
  `msedge --profile-directory=...` により対象プロファイルのウィンドウを起動/前面化する
- ネイティブホストが導入不可の場合、この要件はスコープ外とし、Edge ネイティブの
  プロファイル切替 UI を使う運用にフォールバックする

## 3. 非機能要件

- Manifest V3 準拠。配布は Edge Add-ons ストア公開またはグループポリシー配布
  (組織管理の環境では拡張機能の許可リスト設定に依存 — 導入先で要確認)
- 権限は最小限: `sidePanel`, `tabs`, `tabGroups`, `storage`, `bookmarks`, `favicon`。
  FR-5 採用時のみ `nativeMessaging` を追加
- タブ 200 本規模でもサイドバー描画が遅延しないこと

## 4. 主要な設計判断: Space の実装方式

| 方式 | 内容 | 長所 | 短所 |
|---|---|---|---|
| A. タブグループ方式 | Space = `tabGroups`。切替で他グループを折りたたみ | 1ウィンドウで完結、Arc に近い | 非アクティブ Space のタブも水平タブバーに残る(折りたたみ表示) |
| B. ウィンドウ方式 | Space = ウィンドウ。切替でウィンドウを前面化 | 分離が明確、Profile 連動(FR-5)と整合的 | ウィンドウ管理が煩雑 |
| C. 保存・復元方式 | 切替時に URL リストを保存してタブを閉じ、復元時に開き直す | タブバーが常にきれい | フォーム入力中の状態等が失われる |

**採用方針: A を基本、Profile 付き Space のみ B のハイブリッド。**

## 5. 拡張では超えられない壁(受け入れる制約)

1. **ネイティブの水平タブバーは消せない** → Edge の「垂直タブ」を有効化してタブバーを
   最小化する運用併用を推奨
2. **プロファイル切替はネイティブホストなしでは不可能**(拡張 API の仕様。
   Chromium の Profile Extension API は提案止まりで未実装)
3. 拡張・Pin データはプロファイルごとに分離される。プロファイル横断で Space 一覧を
   出すには、各プロファイルに拡張を入れ、ネイティブホスト経由の状態共有が必要

## 6. 代替案の検討

Edge には垂直タブ・Workspaces(Arc の Space 相当)・プロファイルがネイティブ搭載されている。
「垂直タブ + Workspaces + 手動プロファイル切替」で運用がまわるなら、拡張開発は
FR-1〜3(Pin サイドバー+プレビュー)だけに絞れる。まずネイティブ機能で1週間運用してみて、
足りない部分だけ拡張で作るのが投資対効果として最良。

## 7. 未確定事項(着手前に要回答)

1. 利用環境のポリシー: 拡張のサイドロード/ストア外配布は可能か。Native Messaging
   ホスト(exe/スクリプト)のインストールは可能か → **FR-5 の成否がここで決まる**
2. 対象 OS(macOS / Windows どちらの Edge か。ネイティブホストの実装が変わる)
3. Edge ネイティブの Workspaces で Space 要件がどこまで満たせるか(まず試用推奨)

## 8. 段階的リリース案

1. **Phase 1 (MVP):** FR-1 + FR-2 — サイドバー + Pin。プレビューは favicon+タイトルのみ
2. **Phase 2:** FR-3(サムネイルキャッシュ)+ FR-4(方式 A の Space)
3. **Phase 3:** FR-5(Native Messaging による Profile 連動)— ポリシー確認後に判断
